# ZueDocs Pagefind Search and Fleet Migration Implementation Plan

## Approved decisions and scope

- ZueDocs remains the centrally maintained documentation architecture, visual system, scaffold, and shared runtime for all downstream documentation sites.
- Use Pagefind, not MiniSearch, as the production search engine.
- Hide Pagefind behind a first-party ZueDocs Astro integration and shared ZueDocs components. Downstream projects must not own Pagefind dependencies, Pagefind configuration, search markup, search scripts, or search styling.
- Search is enabled by default for ZueDocs sites. A future consumer may opt out explicitly, but existing and newly scaffolded projects require no product-specific search configuration.
- Existing projects may receive a one-time migration that registers the ZueDocs Astro integration. After that migration, updating `zuedocs` is the only ongoing search-maintenance action required in each consumer.
- New projects generated by `zuedocs init` must include the integration automatically.
- Index only rendered documentation article content for the first release. Do not index home-page marketing copy, repeated navigation, the docs sidebar, the table of contents, footer copy, page-action menus, or raw `.md` endpoints.
- Use Pagefind's accessible Component UI modal as the behavioral foundation, wrapped and themed by ZueDocs. The public consumer contract must not expose Pagefind-specific markup.
- Production and preview builds must have working search. Plain `astro dev` may omit the search trigger for the first release rather than show a broken control; live dev indexing is explicitly deferred.
- Upgrade all consumers to Astro 7 as part of the one-off migration. The only currently discovered Astro-version outlier is `gg-desktop/apps/site`, which is on Astro 5.17.1.
- Delete `/Users/ashray/code/amxv/computer-mcp` during migration after a final clean-worktree and duplicate-remote check. It is a stale checkout of the same `amxv/zodex` remote and must not receive a separate migration.
- The user explicitly approved RPI-fast implementation after this plan is handed to the lead.

## State of Current System

### ZueDocs package architecture

- `package.json` publishes source Astro components, layouts, styles, types, the `docsEnhancements` browser script, and the scaffold CLI. It currently depends only on Mermaid at runtime and does not export an Astro integration.
- `astro.config.mjs` builds a static site but does not register any ZueDocs integration.
- `src/layouts/BaseLayout.astro` owns the standard HTML shell, theme bootstrap, `SiteHeader`, global main content region, and `SiteFooter`.
- `src/components/SiteHeader.astro` owns brand, primary navigation, mobile docs-menu trigger, and theme toggle. This is the correct shared insertion point for a global search trigger/modal because every standard consumer imports the shared base layout, and the `gg-desktop` outlier still imports the shared `SiteHeader` from its local base layout.
- `src/layouts/DocsPageLayout.astro` owns each article's category, title, description, page actions, and Markdown slot. It is the correct shared seam for Pagefind metadata and searchable-body annotations.
- `src/scripts/docsEnhancements.ts` owns existing client-side docs behavior but is page-focused and imported only by docs routes. Search should remain a separate component/script so it is available globally without coupling Pagefind initialization to code-block, Mermaid, table, and sidebar enhancements.
- `src/styles/global.css` defines the complete light/dark token system, shared header styles, responsive mobile header rules, docs layout, page actions, and content styling. Search must consume these variables rather than introducing a separate visual theme.
- `src/types.ts` exposes `SiteConfig`, `PrimaryNavItem`, and footer types. Search should default on; do not require a new `siteConfig.search` property for adoption. If an opt-out is included, make it optional and backwards compatible.
- `bin/zuedocs.js` writes a minimal Astro project with `astro.config.mjs`, package scripts, local content/data/pages, and imports from ZueDocs. Its generated Astro config currently contains no integration.
- `.github/workflows/release.yml` validates with `bun run check`, `bun run build`, and `npm pack --dry-run`, then automatically bumps one patch version, commits/tags, publishes to npm, and pushes the release commit. Feature code must not manually bump `package.json`.
- There is no dedicated automated test suite today.

### Pagefind and Astro capabilities grounding the design

- Pagefind's Node API can create an index, add all HTML from a generated directory, write the bundle to an arbitrary output path, report indexing errors/page count, and shut down its native service.
- Astro's `astro:build:done` integration hook runs after SSG or SSR production output has completed and supplies `dir`, the resolved build-output URL. Using this hook avoids hardcoding `dist` and is required for `finetuning`, whose Astro output is `docs-dist`.
- Pagefind's Component UI can be installed as a package and provides modal/searchbox primitives, keyboard navigation, focus behavior, sub-results, visible and assistive localization, and WAI-ARIA-oriented defaults.
- Pagefind annotations and/or integration-level selectors can exclude repeated shell/navigation elements and capture article metadata.
- Pagefind requires completed static HTML. The first release will therefore enable search in built/previewed output and omit it from plain dev mode.

## State of Ideal System

### Package ownership

- ZueDocs exports a first-party Astro integration, for example `zuedocs/astro`, whose default export returns an `AstroIntegration`.
- The integration uses `astro:build:done` and Pagefind's Node API to index the resolved Astro output directory and write the static bundle under `<outDir>/pagefind`.
- The integration treats Pagefind errors, zero indexed documentation pages, or failed bundle writes as build failures with concise ZueDocs-prefixed logs.
- ZueDocs owns compatible `pagefind` and `@pagefind/component-ui` dependency versions. Consumers do not declare either package directly.
- ZueDocs exports any integration option types it intentionally supports. Initial options should remain narrow: `enabled?: boolean`, and only add `outputSubdir`, `includeCharacters`, or equivalent if they are required and tested. Product-level configuration is not required.
- The shared `SiteHeader` includes a ZueDocs-owned search component. It renders the Pagefind modal/trigger only for production builds, thereby avoiding a dead or 404ing control during plain `astro dev`.
- The shared component imports Pagefind Component UI through its supported package entrypoints, wraps it in ZueDocs class/data hooks, and supplies ZueDocs copy, result templates if required, and theme styling.
- The shared `DocsPageLayout` emits Pagefind searchable-body and metadata annotations so only the article category/title/description/body are indexed. `DocsPageActions`, repeated shell content, sidebars, and tables of contents are excluded.
- Search is accessible from every built page using a visible header trigger and `Cmd+K`/`Ctrl+K`; it is keyboard navigable, dismissible with Escape/backdrop/close, focus-safe, responsive on mobile, and themed in light and dark modes.
- Search results show document title, matching section when available, category/description context, and a highlighted excerpt. Results link to the matching page or heading anchor.
- Search indexes only rendered `/docs/...` articles. Raw `.md` routes do not create duplicate results.

### Scaffold and ongoing maintenance

- `zuedocs init` generates an Astro config importing and registering the ZueDocs integration. It does not expose Pagefind or require a Pagefind config file.
- Generated package manifests depend only on ZueDocs and Astro-related packages; Pagefind remains transitive and centrally versioned.
- Generated build scripts remain ordinary `astro build` commands because the integration owns post-build indexing.
- A new scaffold passes check/build and contains a working `dist/pagefind` bundle immediately after install/build.
- Updating ZueDocs is sufficient to receive future Pagefind, UI, indexing, styling, and relevance changes.

### Migrated fleet

- Each of the 9 canonical upstream projects imports and registers `zuedocs/astro` once, updates to the released ZueDocs version, updates its native lockfile, and otherwise keeps its docs content/routes/configuration local and unchanged.
- Every canonical project uses Astro 7.
- `finetuning/docs-dist/pagefind` is generated correctly from its custom output directory.
- `gg-desktop/apps/site` remains visually customized through its local base layout but receives the shared search trigger/modal through `SiteHeader`; its MDX docs are indexed from rendered output.
- `zodex` and its Cloudflare deployment contain the Pagefind artifacts before Wrangler runs.
- The stale `/Users/ashray/code/amxv/computer-mcp` checkout is removed only after its clean/duplicate status is revalidated.
- Each repository has its migration committed and pushed without unrelated files, with checks and built-search acceptance evidence recorded in the implementation handoff.

## Cross-provider and cross-consumer requirements

### Search-provider boundary

- Pagefind is the sole provider in scope. Do not implement MiniSearch, Algolia, a provider union, analytics, multisite federation, or a hosted search service in this feature.
- Keep Pagefind behind `zuedocs/astro` and the shared search component so a future provider change does not alter consumer pages.
- Do not put Pagefind credentials or environment variables in consumers; Pagefind remains a local build-time/static-browser system.
- Do not copy or fork Pagefind's internal implementation. Use its documented Node API and Component UI package entrypoints.

### Astro/package-manager compatibility

- Target Astro 7 throughout the fleet after migration.
- The integration must use the `dir` URL supplied by `astro:build:done` and convert it with `fileURLToPath`; never assume `dist`, repository root, or current working directory.
- Validate both Bun-driven Astro builds and the npm-driven `finetuning` build.
- Consumers must not add direct Pagefind dependencies; package resolution and upgrades are centralized in ZueDocs.
- Preserve existing Astro integrations and Vite plugins. In `gg-desktop/apps/site`, append `zuedocs()` alongside `mdx()` and retain the Tailwind Vite plugin.

### Indexing contract

- A rendered article is searchable when it uses `DocsPageLayout`. This is the stable package-level contract across Markdown and MDX consumers.
- Include article title, description, category, headings, paragraphs, lists, tables, and code text.
- Exclude header/navigation, footer, docs sidebar, table of contents, mobile backdrop, page-actions menu, theme controls, and generated scripts/styles.
- Use the current page `<html lang>` value for Pagefind language behavior. This release may retain the existing English default; do not expand into full site internationalization.
- Begin with a conservative developer-doc punctuation policy. Test exact flags, dotted identifiers, underscored names, paths, and natural-language searches before settling `includeCharacters`; do not preserve punctuation speculatively if it harms normal matching.
- Ensure Pagefind sees stable heading IDs produced by Astro and exposes useful sub-results without duplicate page spam.

### Deployment contract

- The output remains a static Astro site. No serverless function, database, API key, or hosted crawler is introduced.
- Pagefind output must be produced before Astro's build command returns so Vercel, Wrangler, and other existing deployment commands capture it automatically.
- The integration must work with default `dist` and custom `docs-dist` output directories.
- Search bundle paths must be root-relative for the currently discovered consumers. Subpath/base support is not required unless an existing consumer build proves it is needed.

### Phase 1 — Build the first-party ZueDocs Pagefind Astro integration

#### Files to read before starting

- `package.json`
- `bun.lock`
- `astro.config.mjs`
- `src/index.ts`
- `src/types.ts`
- `.github/workflows/release.yml`
- Pagefind Node API documentation
- Astro Integration API documentation for `astro:build:done`

#### What to do

1. Add centrally managed dependencies for the Pagefind Node API and Pagefind Component UI. Use compatible release versions and keep them in ZueDocs, not downstream manifests.
2. Add an exported integration module, preferably `src/integrations/astro.ts`, and expose it as `zuedocs/astro` in `package.json`.
3. Return a named Astro integration such as `amxv:zuedocs` with an `astro:build:done` hook.
4. In a separately testable helper:
   - convert the hook's `dir` URL to an absolute filesystem path;
   - create a Pagefind index with documented Node API options;
   - add the generated HTML directory with an HTML-only glob;
   - use ZueDocs-controlled indexing options and conservative special-character support;
   - inspect `errors` and indexed `page_count`;
   - write the bundle to `<resolved-out-dir>/pagefind`;
   - inspect write errors;
   - close/delete the Pagefind service/index in `finally` cleanup;
   - log indexed page count and bundle location through Astro's integration logger.
5. Fail the build on Pagefind create/add/write errors. Decide explicitly whether zero indexed pages is a hard failure; for the ZueDocs scaffold and known consumers it should be, because a configured ZueDocs docs site without searchable pages indicates broken annotations or routes.
6. Keep initial options internal and defaults-first. If an integration option type is necessary, export it from `src/index.ts`/`src/types.ts`, make it backward compatible, and do not require consumers to pass anything.
7. Register the integration in ZueDocs' own `astro.config.mjs` so the package site dogfoods the exact published path.
8. Add a focused test around the indexing helper using temporary generated HTML fixtures. Assert that ignored shell text is absent, article text is searchable/indexed, errors fail, a non-default output path works, and expected Pagefind bundle entry files are written.
9. Add the test script to package validation without weakening existing check/build/pack gates.

#### Validation strategy

- Run the focused integration/helper tests.
- Run `bun run check`, then `bun run build` serially.
- Assert `dist/pagefind` exists and contains the Pagefind runtime, metadata, index fragments, and any assets required by the selected Component UI import mode.
- Run a local static preview and verify direct requests to the Pagefind bundle succeed.
- Run `npm pack --dry-run` and verify the integration source/export and required component code are included.
- Validate the helper against a temporary non-`dist` output directory to cover `finetuning` before fleet migration.

#### Risks / fallbacks

- **Risk:** The Pagefind Node API/native binary fails under Bun even though it works under Node. **Fallback:** run the integration through the documented Node API from Astro's process and validate under both Bun and npm before proceeding; if Bun incompatibility is confirmed, use the Pagefind CLI binary through a package-resolved executable only as a documented fallback, not a networked `bunx` call.
- **Risk:** Component UI and engine versions drift. **Fallback:** pin or constrain them to a tested compatible release pair in ZueDocs and upgrade them together.
- **Risk:** Pagefind indexes raw `.md` outputs or unrelated HTML. **Fallback:** use the HTML glob plus article annotations/exclusions and assert result URLs in tests.
- **Risk:** `astro:build:done` runs for non-static output. **Fallback:** inspect resolved Astro config during integration setup/done and fail with a clear message or skip only when explicitly disabled; all current consumers are static after migration.

#### Validation strategy

  - a body-only phrase;

  - a guaranteed no-result query.
- Inspect returned URLs and verify raw `.md` endpoints, home pages, sidebars, footer copy, and page-action labels do not appear as results.
- Keyboard-test trigger, modal focus, arrow keys, Enter, modified Enter, Tab, Escape, and focus restoration.
- Test mobile viewport behavior and both light/dark/system theme states.
- Verify `astro dev` has no dead search trigger or asset 404 caused by ZueDocs.

#### Risks / fallbacks

- **Risk:** Importing Component UI from a package fails when the Astro component is consumed from `node_modules`. **Fallback:** use Pagefind's generated Component UI assets with a ZueDocs loader that injects the stylesheet/module from `/pagefind`, but preserve self-contained behavior for local base-layout consumers.

- **Risk:** `data-pagefind-body` on article layouts causes docs-index pages to be omitted. **Fallback:** accept docs-index omission for v1 because the requirement is searchable article content; do not broaden scope to marketing/index pages merely to increase page count.
- **Risk:** Search controls overcrowd the existing mobile header. **Fallback:** use the compact icon trigger on small screens and preserve docs-menu/theme/navigation priority.

   - no Pagefind asset 404s, console errors, or stale indexes exist.

#### Validation strategy

- Automated: package tests/check/build/pack; each consumer's native check/build; presence of Pagefind bundle and direct asset access.
- Functional: representative search-query matrix on local preview and live deployment.
- UX/accessibility: keyboard-only, mobile, focus management, reduced motion, both themes, no-result/loading states.
- Architectural: no direct consumer Pagefind dependencies/config/files; only `zuedocs()` integration registration plus package/lock updates; no duplicated search components/scripts/styles.
- Release: npm version, git tag, ZueDocs main, consumer lockfiles, and live deployments all agree.

#### Risks / fallbacks

- **Risk:** Live alias remains stale after a successful push. **Fallback:** inspect deployment HTML/assets and trigger an explicit production deployment where the repository's existing workflow permits it.
- **Risk:** CDN caches an old Pagefind manifest/index. **Fallback:** verify emitted filenames/cache headers, redeploy the fresh build, and correct shared asset-loading/version behavior in ZueDocs if necessary.
- **Risk:** One project exposes unusual docs routes not found during inventory. **Fallback:** adjust shared annotations/index filtering if the route still uses the ZueDocs contract; add consumer configuration only if the difference is genuinely product-specific and cannot be derived.
- **Risk:** A project exceeds expected client/query performance. **Fallback:** tune Pagefind metadata, filters, and result limits centrally; the current fleets are all below the user's stated ~100-page ceiling.

## Explicit non-goals

- No hosted Algolia/other search service.
- No MiniSearch fallback in production.
- No search analytics or query collection.
- No cross-site/federated search across multiple products.
- No local copies of Pagefind configuration or search UI in consumer repositories.
- No broad rewrite of consumer docs routes, data models, or product copy.
- No live Pagefind index during plain `astro dev` in the first release.
- No repository-specific search themes; all consumers use the shared ZueDocs theme behavior.

- ZueDocs package checks/build/tests/pack pass and the automated release publishes the feature version.
- The stale `computer-mcp` duplicate checkout is safely removed.
- All 9 canonical upstream projects are on Astro 7, reference the released ZueDocs version, register `zuedocs()`, pass native checks/builds, emit Pagefind bundles, and have pushed migrations.

- Live search is verified across the fleet with no duplicated shared implementation.
- A deep RPI-fast acceptance reviewer explicitly reports `CLEARED` before final lead completion.
